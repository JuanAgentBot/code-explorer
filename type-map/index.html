How it works

Analysis

Your code is parsed in the browser using the TypeScript Compiler API (typescript 5.8, loaded from CDN). No server, no uploads.

  1. ts.createSourceFile parses the input into an AST (Abstract Syntax Tree).
  2. First pass collects all top-level declaration names (interfaces, type aliases, classes, enums) into a set.
  3. Second pass walks each declaration: extracts members (properties, methods, enum values), and detects relationships by checking heritage clauses (extends, implements) and scanning member type annotations for references to known names.
  4. Edges are deduplicated. The result is a graph of nodes (declarations) and edges (relationships).

Rendering

The diagram is hand-built SVG with no chart library. Nodes are placed using a simple grid layout (row-major, max 4 columns).

  • Each declaration becomes a box. Border color indicates kind: purple = interface, cyan = type alias, pink = class, orange = enum.
  • Members are listed inside the box with name and type annotation. Methods show () suffix.
  • Edges are cubic Bezier curves between box centers with SVG marker arrowheads.

Limitations

  • Only top-level declarations are extracted. Nested types inside function bodies or namespaces are ignored.
  • Reference detection is name-based string matching on type annotation text, not full type resolution. It can miss generics like Array<User> and may false-positive on substring matches.
  • Layout is a static grid. No edge routing, no overlap avoidance. Large graphs get crowded.

Stack

  • Parser: TypeScript Compiler API (syntax-only mode via createSourceFile, no type checker needed)
  • Rendering: Raw SVG string construction. No D3, no Mermaid, no chart library.
  • Layout: Custom row-major grid. Box sizes computed from member count and text length.
  • Build: Vite. 17 KB app bundle + TypeScript compiler from CDN.
